FINAL PROJECT
Ukulele Tuner

Authors Megha Mishra & Kaiken Ancheta
Video Demo
1

Description

Musical instruments such as the ukulele must be tuned properly in order to produce accurate and harmonious sound. To do this, each string must be tightened such that it vibrates at the frequency that matches a given tuning type (i.e Standard Tuning: G4, C4, E4, A4).

The goal of this project was to create a beginner-friendly and highly customizable ukulele tuner. The system uses the INMP441 I2S MEMS microphone to capture sound and uses the YIN pitch detection algorithm to determine the fundamental frequency. The detected frequency is then compared to the frequency of the current string that the user is tuning to determine whether it is within the user-decided sensitivity threshold or not. To assist beginners throughout the tuning process, the display prompts the user to pluck each string in order and indicates whether the tuning peg should be turned clockwise or counterclockwise. In addition, state changes can be made using the AWS Device Shadow, where the user can change parameters such as tuner sensitivity, tuning type, and various display settings.

Design

2.1

Functional Specification

The figure below depicts a high-level state diagram that demonstrates the control flow of the system.

IDLE: This state is reached when the system is powered ON. The system establishes internet connection and displays an initialization message to the user. In this state, the system waits to detect device tilt to enter the next state.

AWS UPDATE: Once the device detects tilt, the system sends a GET request to check if the desired state has been updated from AWS. The system then parses through the data to update the parameters in the program accordingly. Once the data has been gathered, the system displays a Type Menu for the user and waits for the user to select a tuning type using the remote.

TYPE SELECTION: When the user presses a remote button, the outputFull is set and the system uses the decode() function to select a set of note names and frequencies for the next stage. The device starts sampling data and computing the RMS to check for sound above the user-decided threshold level.

SIGNAL PROCESSING: If the detected sound surpassed the sound amplitude threshold, the detected signal is processed to determine the fundamental frequency. This frequency is then compared to the ideal frequency of the current string to check for deviation.

DISPLAY OUTPUT: The system displays the current note name of the string being tuned as well as a visual indication of how sharp or flat the detected frequency is. Once the information is displayed to the user, the state transitions to the SIGNAL PROCESSING state until the tuningComplete flag is reset, at which point the system returns to idle.

2.2

System Architecture

The figure below shows a high-level block diagram of the system architecture for this project.

As demonstrated above, the sound from the instrument is detected by the INMP441 microphone module and is communicated to the CC3200 microcontroller using the I2S protocol. The AT&T S10-S3 remote sends a signal to the IR receiver circuit, which produces an output signal that is connected to a GPIO input pin on the CC3200 board. The built-in BMA222 accelerometer is enabled through I2C in order to detect tilt. The Adafruit OLED display is connected to the CC3200 board via SPI protocol. Lastly, a host device provides power to the CC3200 board through a Micro-USB to USB-A cable, as well as wireless Wifi connectivity. The host device is also set up with an Amazon Web Services (AWS) account, where the device shadow can be remotely updated and accessed with a GET request.

3

Implementation

The first step in the implementation process was to accurately detect frequency from a sound source. For this project, the INMP441 microphone was used for its high signal-to-noise ratio (SNR). To collect samples, I2S was first initialized in main and the MAP_I2SDataGet() function was used to fill a buffer with I2S samples.

The YIN method was used to detect frequency while in the time domain. First, the RMS of the detected samples was calculated to check if the sound was, on average, above a user detected sound amplitude threshold. Then, the DC component of the detected signal was removed. In the YIN frequency detection algorithm, the first step was to calculate the squared differences between the original signal and a delayed version and store it in the fDiff[] buffer. Then, the contents of the buffer are normalized by their cumulative mean to eliminate common harmonic frequencies such as octaves. Then, a frequency window was dynamically calculated within 5 percent of the current string's frequency. Within the lo and hi bounds, the deepest local minimum was found. Lastly, parabolic interpolation was used to more precisely determine the period of the detected signal without increasing sampling rate. This frequency was returned to main and passed into the report_tuning() function, where the deviation in cents between the measured and target frequency was calculated.

The next step in the implementation process was to use the accelerometer built into the CC3200 board to detect that the board is tilted towards the instrument to ensure that the detected samples were as clean as possible. First, I2C was configured in main. Then, a do-while loop was used to read the accelerometer x and y values, type-cast those values to an 8-bit integer value, and compare them to threshold tilt values. To ensure that the x-tilt was in correct direction, the conditional statement was written to check for x-tilt in the negative direction below a threshold value determined through trial and error, as well as check for near-0 y-tilt.

After verifying the accelerometer logic, the next step in the implementation process was to establish proper communication with Amazon Web Services (AWS). In main, after the accelerometer detects that the board is tilted, tls_connect() is called to establish a connection to AWS. After a successful connection, http_get() is called to retrieve the current device shadow state. http_get() sends an HTTP GET request to the AWS device shadow and parses the response. The request is assembled by concatenating the GET header, host header, connection header, and line terminator into a buffer, and this request is then sent with sl_Send(). The response is received into acRecvBuff with sl_Recv(). This response is then parsed through by first using strstr() to locate the "desired" block in the JSON message. From there, it similarly parses the following fields: sensitivity, show_cents, cents_tolerance, and dark_mode. After these fields, it parses over the fields for "button1," "button2," and "button3." For each button, it reads four "note" strings and four "freq" floats, and stores them in their respective arrays. These values are used to update their corresponding global variables, so that the program will use the user's custom settings. Once the shadow data is received, the socket is closed with sl_close().

Once the AWS connection was successful, the full system was connected to an IR receiver circuit so that the tuning type could be selected via remote button press. First, GPIO falling-edge interrupts and the built-in timer were configured in main. In the GPIO interrupt handler, the width of each pulse was calculated by checking the value of the countdown timer and subtracting from the last stored value. If a wide pulse was detected, the variable IR_Output was shifted left. If a narrow pulse was detected, IR_Output was shifted left and appended with a 1. Before filling IR_Output, the start flag must be set to ensure that the signal is always read from the beginning. After each detected pulse, the count variable is incremented. Once 48 pulses are detected, the outputFull flag is set. In main, once the outputFull flag is set, the decode() function is called, where a switch-case statement is used to check the value of IR_Output and select which array of notes and frequencies to use.

After the remote control logic was verified, the OLED display was configured. The OLED display is used for providing instructions, information, and feedback to the user during the tuning process. The main functions used are Adafruit_init to initialize the display, setCursor() to change the position of the cursor for drawing, fillRect() and drawRect() to draw filled and unfilled rectangles of different sizes, setTextSize() to change the font size of drawn characters, Outstr() to print a string, setTextColor() to change the display colors for either dark or light mode, drawCircle() to draw a circle around the displayed note, and fillTriangle() to draw a triangle to be used as an arrow. Once the device is ready, the display will prompt the user to tilt the CC3200 to start the tuning process. This will then retrieve the device shadow data from AWS, after which the display will tell the user to select between three different number buttons: 1, 2, and 3. The display also shows which notes correspond to each of these buttons. Once the tuning begins, the display will show the current string being tuned in the center, surrounded by a circle. The top of the OLED has 7 rectangles. Depending on the detected cents, one of the rectangles will be filled with a color, red, orange, yellow, or green, to indicate how far away the string is from being in tune. When the string is flat or sharp, an arrow is drawn on the circle to show the user which direction to turn the peg. If the user enables displayCents, the bottom of the display will show the detected cents, and if the string is flat, sharp, or in tune.

Challenges

One of the challenges we faced during this project was connecting the IR receiver circuit to the rest of the system to enable selection between multiple tuning types. A while loop was used to wait until the receiver output variable was filled with a value corresponding to the pulses detected by the receiver. When running the program, we realized that this loop was impossible to break out of. After building an isolated IR circuit, we noticed that we still were not able to detect a signal, proving that the issue was not with our logic. We decided to use the Logic Analyzer to determine whether or not we were able to receive a signal from the receiver to the microcontroller. Using the datasheet, we tried all available GPIO pins and found that only PIN 50 was capable of detecting the IR signal. Due to the fact that PIN 50 was being used as an I2S line, we checked the datasheet to determine find a replacement pin for the current purpose of the pin. Once we resolved this issue, we were able to detect the button presses successfully.

Another issue we faced was harmonic interference. Although we were able to detect the G and C strings with relatively high accuracy, we noticed that we were only able to detect either E or A accurately at any given point. This is because E and A share a strong harmonic due to the fact that they are a perfect fourth apart. To resolve this issue, we decided to update our design to prompt the user to tune each string in order so that a bandpass filter could be applied for each string.

Another issue was figuring out how to parse through the button states of the JSON message in http_get(). This is because "button1," "button2," and "button3" were nested further into their own blocks, each with "note1" to "note4" and "freq1" to "freq4." These field names are shared amongst the three button states, so only finding the first occurrence of each using a single strstr() would not work as it did previously. The fix involved using two nested for loops to first iterate through the three button states and then iterate through the four notes and frequencies for each. This way, notes and frequencies wouldn't be mismatched to the wrong button.

We also encountered an issue in drawing triangles on the OLED display. Originally, we wanted multiple arrows to show the clockwise or counterclockwise direction to make it clearer. The plan was to have one arrow on the left, right, and bottom parts of the circle pointing towards a certain direction. However, after much trial and error, we found that we were unable to draw a sideways pointing triangle, leading us to scrap the idea of the bottom triangle. There were also issues in drawing upwards-pointing arrows on either the left or right part of the circle. When these triangles were drawn, their right sides would be cut off. We were unable to fix this issue, so the final design only uses a single downwards-pointing arrow on the left and right sides.

Future Work

While the current system successfully detects and analyzes the pitch of ukulele strings, several improvements can be made in future work. One potential enhancement is to create a 3D-printed enclosure for the device to conceal wires, as well as a mechanism to attach the tuner to the instrument.

Another potential improvement is to expand the bandwidth of the tuner to support other musical instruments, such as the guitar, violin, or bass. Currently, only 3 of the many available remote buttons are used to select the tuning type, but other buttons could be used to select instrument and display preferences.

We would also like to make hardware improvements to improve the speed and reliability of the system. Our power supply is currently provided by the CC3200 board, which is spotty and creates a flashing effect on the OLED display. In addition, although our chosen microphone works in a completely noise-free setting, a better microphone module can be used to further eliminate noise and allow the tuner to be used reliably in any setting.
